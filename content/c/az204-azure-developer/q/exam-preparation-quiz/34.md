    - id: "answer1"
      title: "Test application code by rebooting all nodes in the test environment"
      correct: false
      explain: "Rebooting nodes is disruptive and doesn't address the root causes of Redis overload. While it might temporarily clear the issue, it doesn't implement any preventive measures like retry logic or memory management improvements. Rebooting is a last resort, not a proactive solution."
    - id: "answer2"
      title: "Modify the maxmemory policy to evict the least frequently used keys out of all keys"
      correct: false
      explain: "Changing the eviction policy (like using allkeys-lfu for least frequently used) affects which keys are removed when memory is full, but doesn't prevent overload or handle connection issues. This is about memory management strategy, not resolving current overload conditions or improving resilience."
    - id: "answer3"
      title: "Configure client connections to retry commands with exponential backoff"
      correct: true
      explain: "Implementing retry logic with exponential backoff helps clients gracefully handle transient failures and connection issues during overload scenarios. This prevents cascading failures, reduces request storms, and allows the Redis cache time to recover from temporary spikes in load."
    - id: "answer4"
      title: "Test application code by purging the cache in the test environment"
      correct: false
      explain: "Purging the cache in test doesn't address production overload issues and could actually make things worse by causing a cache stampede when all keys need to be reloaded. This is a testing technique, not a solution for preventing or resolving overload conditions."
    - id: "answer5"
      title: "Increase the maxmemory-reserved and maxfragmentationmemory-reserved values"
      correct: true
      explain: "Reserving more memory for Redis operations prevents memory pressure that causes performance degradation and connection issues. These settings ensure Redis has buffer memory for background operations like persistence, replication, and defragmentation, reducing the likelihood of overload conditions."
